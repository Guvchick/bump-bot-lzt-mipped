# ТЗ / промпт: Telegram-бот на Go для авто-апа тем (Lolzteam + Mipped)

> Скопируй этот файл целиком в кодинг-агента (Claude Code / Cursor / и т.п.) как задание.
> Раздел «Справочник API» содержит проверенные данные — **используй их как есть, не выдумывай эндпоинты**.

---

## 1. Что нужно построить

Telegram-бот на **Go**, который автоматически поднимает («апает») мои темы на двух форумах:

- **Lolzteam** (`lolz.live`) — через **официальный REST API** по Bearer-токену.
- **Mipped** (`mipped.com`) — через **сессию (логин/пароль)** + штатную кнопку «Поднять тему» (это встроенная бесплатная функция форума с лимитом раз в 24 ч; на платных тарифах форум сам продаёт авто-ап 6/24 раза в день — мы делаем бесплатный аналог в рамках лимита).

Бот управляется через **Telegram-панель** (inline-кнопки): аккаунты, темы, статистика, логи, настройки. Только для меня (whitelist по Telegram ID).

---

## 2. Стек и зависимости

- **Go 1.22+**
- Telegram: `gopkg.in/telebot.v4` (или `github.com/go-telegram/bot` — на твой выбор, но панель на inline-кнопках с FSM)
- HTTP: стандартный `net/http` + `net/http/cookiejar` (для сессий Mipped)
- Парсинг HTML: `github.com/PuerkitoBio/goquery` (вытаскивание CSRF-токена и статистики со страниц Mipped)
- БД: **SQLite** через `modernc.org/sqlite` (чистый Go, без CGO) — для одного пользователя этого хватит.
  Сделай слой хранилища интерфейсом `Storage`, чтобы при желании заменить на Postgres (`pgx`) — я с ним работаю.
- Конфиг: `.env` через `github.com/joho/godotenv`
- Шифрование секретов: `crypto/aes` + `crypto/cipher` (AES-GCM)
- Логи: `log/slog`

│   │   └── mipped.go  # клиент Mipped (сессия + /up + парсинг)
│   ├── scheduler/     # планировщик апов (per-thread, с persist next_bump_at)
│   └── tg/            # Telegram-панель: хендлеры, клавиатуры, FSM
├── migrations/        # SQL-схема

Ключевая абстракция:

```go
type Forum interface {
    // Поднять тему. Возвращает результат и, если форум просит подождать,
    // длительность до следующей возможной попытки.
    Bump(ctx context.Context, acc Account, t Thread) (BumpResult, error)
    // Статистика темы (просмотры, ответы, заголовок).
    ThreadStats(ctx context.Context, acc Account, t Thread) (ThreadStats, error)
    // Проверка валидности аккаунта (токен жив / сессия активна).
    CheckAuth(ctx context.Context, acc Account) error
}

type BumpResult struct {
    OK        bool
    Message   string         // текст ответа форума (для лога)
    RetryAfter time.Duration // если > 0 — форум просит подождать столько
}
```

## 4. Модель данных (БД)

**accounts**
- `id` PK
- `forum` TEXT — `"lolz"` | `"mipped"`
- `label` TEXT — человекочитаемое имя
- `secret_enc` BLOB — зашифрованный секрет:
  - для lolz: API-токен
  - для mipped: JSON `{"login": "...", "password": "..."}`
- `session_enc` BLOB NULL — закэшированные cookies Mipped (чтобы не логиниться каждый раз)
- `proxy` TEXT NULL — `socks5://user:pass@ip:port` (опционально, особенно для Mipped)
- `status` TEXT — `ok` | `auth_failed` | `unknown`
- `created_at`, `updated_at`

**threads**
- `id` PK
- `account_id` FK
- `forum` TEXT
- `thread_ref` TEXT — для lolz: числовой `thread_id`; для mipped: полный URL темы (из него парсим slug+id)
- `title` TEXT NULL — подтягивается из статистики
- `interval_sec` INTEGER — желаемый интервал апа (по умолчанию из настроек)
- `enabled` BOOL — включён ли автоап
- `next_bump_at` TIMESTAMP NULL — когда следующая попытка
- `last_bump_at` TIMESTAMP NULL
- `created_at`, `updated_at`

**bump_log**
- `id` PK
- `thread_id` FK
- `at` TIMESTAMP
- `ok` BOOL
- `message` TEXT — текст ответа / ошибки
- `next_at` TIMESTAMP NULL

**stats_snapshots** (для графика роста)
- `id` PK
- `thread_id` FK
- `at` TIMESTAMP
- `views` INTEGER NULL
- `replies` INTEGER NULL

**settings** (key-value)
- `default_interval_sec` (например 86400 для mipped, 14400 для lolz — у lolz свой кулдаун)
- `jitter_sec` — рандомный разброс времени апа (например 0–1800 сек), чтобы не апать секунда-в-секунду
- `stats_poll_sec` — как часто снимать снапшот статистики (например раз в час)

## 5. Справочник API (ПРОВЕРЕНО — использовать дословно)

### 5.1 Lolzteam (официальный API)

- **Base URL:** `https://prod-api.lolz.live`
- **Авторизация:** заголовок `Authorization: Bearer <TOKEN>`
  Токен я получаю в `https://lolz.live/account/api`.
- **Заголовки:** `Accept: application/json`
- **Минимальная задержка между запросами:** делай настраиваемый `delay_min` (≈3 сек), официальные клиенты держат задержку по умолчанию. Не долбить.

**Поднять тему (bump):**
```
POST https://prod-api.lolz.live/threads/{thread_id}/bump
Authorization: Bearer <TOKEN>
```
Ответ — JSON. Если рано — вернётся ошибка с описанием (по аналогии с XenForo: поле `errors`/`error`). На ошибку — backoff (см. §7).

**Статистика темы:**
```
GET https://prod-api.lolz.live/threads/{thread_id}
Authorization: Bearer <TOKEN>
```
В ответе объект `thread` с полями вида `thread_view_count`, `thread_post_count`, `thread_title`, `thread_is_visible` и т.д. — распарси просмотры/ответы/заголовок.
(Это XenForo-стиль ответа: смотри фактический JSON и достань нужные ключи; имена могут быть `view_count`/`post_count` либо с префиксом `thread_` — обработай оба варианта.)

**Проверка авторизации:**
```
GET https://prod-api.lolz.live/users/me        (или /users/{id})
```
200 = токен жив.

> Полная дока (для AI): индекс эндпоинтов в OpenAPI/Markdown — `https://lolzteam.readme.io/llms.txt`.
> Питон-клиент-референс структуры: репозиторий `AS7RIDENIED/LOLZTEAM`, файл `Documentation/Forum.md`.

### 5.2 Mipped (XenForo, своего публичного API НЕТ — работаем через сессию)

Mipped — это XenForo 2.x. Публичный REST API для рядовых пользователей закрыт (ключи генерит только админ), поэтому действуем как браузер.

#### Шаг A — Логин (получение cookies)
1. `GET https://mipped.com/f/login/` — забрать страницу и cookies, распарсить `_xfToken`.
2. `POST https://mipped.com/f/login/login`
   Content-Type: `application/x-www-form-urlencoded`
   Тело:
   ```
   login={login}
   password={password}
   remember=1
   _xfRedirect=https://mipped.com/f/
   _xfToken={token_из_шага_1}
   ```
3. После успеха в cookiejar будут `xf_user`, `xf_session`, `xf_csrf`. **Сохрани cookies в `accounts.session_enc`** и переиспользуй; перелогинивайся только если сессия протухла (см. ниже).

#### Шаг B — Свежий CSRF-токен перед апом
`_xfToken` имеет формат `unixtime,hash` и привязан к сессии/времени — его нельзя взять из cookie. Перед каждым апом:
1. `GET https://mipped.com/f/threads/{slug}.{thread_id}/` (страница темы).
2. Достать актуальный токен из HTML. Он лежит в одном из мест (проверь по порядку через goquery):
   - атрибут `<html ... data-csrf="...">`
   - или скрытый input `input[name="_xfToken"]`
   - или в JS: `XF.config.csrf` (regex по тексту страницы).

#### Шаг C — Ап темы (ПРОВЕРЕНО через DevTools)
```
POST https://mipped.com/f/threads/{slug}.{thread_id}/up
Content-Type: application/json
X-Requested-With: XMLHttpRequest
Origin: https://mipped.com
Referer: https://mipped.com/f/threads/{slug}.{thread_id}/
Cookie: <сессионные куки>
User-Agent: <реалистичный, см. §7>
```
Тело (JSON):
```json
{
  "_xfRequestUri": "/f/threads/{slug}.{thread_id}/",
  "_xfResponseType": "json",
  "_xfToken": "{свежий_токен}",
  "_xfWithData": 1
}
```

**Парсинг ответа (ПРОВЕРЕНО):**
- Успех:
  ```json
  { "status": "ok", ... }
  ```
- Рано (лимит 24ч):
  ```json
  {
    "status": "error",
    "errors": ["Нужно подождать 0 дней, 23 часов, 57 минут и 38 секунд чтобы поднять тему"],
    "errorHtml": { ... }
  }
  ```
  HTTP-код при этом **403** — это нормально, не считать фатальной ошибкой. Нужно **распарсить оставшееся время** из текста и поставить `next_bump_at = now + remaining + буфер (например +60 сек)`.

  Regex под русский текст (дни/часы/минуты/секунды могут отсутствовать):
  ```
  (?:(\d+)\s*дн)?.*?(?:(\d+)\s*час)?.*?(?:(\d+)\s*минут)?.*?(?:(\d+)\s*секунд)?
  ```
  Безопаснее: вытащить все числа в порядке появления и сопоставить с найденными единицами («дн», «час», «минут», «секунд»).

#### Признаки протухшей сессии
Если страница темы редиректит на логин, или в ответе на `/up` приходит ошибка про авторизацию / «войдите» / приходит HTML логина вместо JSON — пометить `status=auth_failed`, выполнить повторный логин (Шаг A) и повторить ап один раз.

#### Статистика Mipped (best-effort скрейпинг)
На странице темы через goquery достать:
- заголовок темы (`<h1>` / og:title),
- кол-во ответов и просмотров (в XenForo обычно в блоке `.pairs`/`.message-attribution` или в `<dl>` со словами «Ответы»/«Просмотры»; разметка может отличаться — **сначала загляни в реальный HTML страницы и подбери селекторы**).
Если что-то не парсится — сохраняй `NULL`, не падай.

## 6. Планировщик апов

- На старте загрузить все `enabled` темы, для каждой вычислить задержку до `next_bump_at` (если в прошлом — выполнить почти сразу, с маленьким случайным сдвигом).
- Реализация: на каждую тему свой управляемый таймер (горутина + `time.Timer`), либо единый цикл-тикер (раз в 30 сек) который выбирает темы с `next_bump_at <= now`. Второй вариант проще и надёжнее при многих темах — предпочти его.
- После попытки апа:
  - **OK** → `last_bump_at=now`, `next_bump_at = now + interval + rand(jitter)`, запись в `bump_log` (ok=true).
  - **RetryAfter > 0** → `next_bump_at = now + RetryAfter + буфер`, запись в `bump_log` (ok=false, message=текст).
  - **Сетевая/прочая ошибка** → экспоненциальный backoff (например 1м → 5м → 15м, кап 1ч), запись в лог.
- Между апами разных тем/аккаунтов — соблюдать `delay_min`, чтобы не слать пачку запросов одновременно.
- Отдельный воркер статистики: раз в `stats_poll_sec` снимать снапшот по каждой активной теме и писать в `stats_snapshots`.

## 7. Анти-бан / аккуратность

- **User-Agent**: задать реалистичный десктопный UA (например свежий Firefox/Chrome) и держать его постоянным на аккаунт. Не использовать пустой/«Go-http-client».
- **Джиттер по времени**: никогда не апать ровно в HH:00:00 — добавлять случайные минуты (`jitter_sec`).
- **Задержки между запросами**: глобальный `delay_min`.
- **Прокси на аккаунт** (опционально): прокинуть через `http.Transport`/`golang.org/x/net/proxy` (socks5). Полезно для Mipped.
- **Не флудить ретраями**: при ошибке — backoff, а не цикл в лоб.
- Lolzteam — только официальный API (это легально и предусмотрено форумом). Mipped — только штатная кнопка апа в рамках суточного лимита, который сам сервер и контролирует.

## 8. Telegram-панель

**Доступ:** бот реагирует только на ID из `OWNER_IDS` (через запятую в `.env`). Остальным — игнор/«нет доступа».

**Главное меню** (inline-кнопки):
```
[ 👤 Аккаунты ]  [ 📌 Темы ]
[ 📊 Статистика ] [ 🧾 Логи ]
[ ⚙️ Настройки ]
```

**👤 Аккаунты**
- Список аккаунтов: `label`, форум, статус (🟢 ok / 🔴 auth_failed), кол-во тем.
- `➕ Добавить`:
  - выбрать форум (Lolz / Mipped),
  - Lolz: запросить токен (FSM, одно сообщение → сохранить зашифрованно → сразу `CheckAuth`).
  - Mipped: запросить логин, затем пароль (FSM) → сохранить зашифрованно → выполнить тестовый логин.
  - **После сохранения секрета — удалить сообщение пользователя с токеном/паролем из чата** (`bot.Delete`), чтобы не светился в истории.
- По аккаунту: `🔄 Проверить`, `🗑 Удалить`, `✏️ Прокси`.

**📌 Темы**
- Список тем: заголовок (или ref), форум, аккаунт, статус автоапа (✅/⏸), `next_bump_at` (относительно: «через 3 ч 12 м»).
- `➕ Добавить тему`: выбрать аккаунт → ввести `thread_id` (Lolz) или URL темы (Mipped) → задать интервал (или взять дефолт) → сразу подтянуть заголовок через `ThreadStats`.
- По теме: `⬆️ Апнуть сейчас` (ручной бамп вне расписания, с учётом ответа форума), `⏯ Вкл/Выкл автоап`, `⏱ Интервал`, `📊 Статистика темы`, `🗑 Удалить`.

**📊 Статистика**
- По теме: текущие просмотры/ответы + дельта за 24 ч/7 дней (из `stats_snapshots`), кол-во успешных апов, время последнего апа.
- Простой текстовый «график» роста (например спарклайн символами ▁▂▃▅▇ или ASCII) — без внешних либ.
- Сводка по всем темам: сколько активно, сколько ждут, сколько в ошибке.

**🧾 Логи**
- Последние N записей `bump_log` (по теме или общие): время, тема, ✅/❌, текст ответа, когда следующий ап.

**⚙️ Настройки**
- Дефолтный интервал, джиттер, период снятия статистики, глобальная задержка между запросами.

**Уведомления** (опционально, флагом в настройках): слать в чат при переходе аккаунта в `auth_failed` и при N подряд неудачных апах темы.

## 9. Конфиг (`.env.example`)

```
TELEGRAM_BOT_TOKEN=
OWNER_IDS=123456789
DB_PATH=./bot.db
ENCRYPTION_KEY=            # 32 байта base64 — ключ для AES-GCM (секреты в БД)
DEFAULT_INTERVAL_SEC=86400 # дефолт для mipped
LOLZ_INTERVAL_SEC=14400    # дефолт для lolz (учесть кулдаун форума)
JITTER_SEC=1800
STATS_POLL_SEC=3600
REQUEST_DELAY_MS=3000
LOG_LEVEL=info
```

Секреты (токены, пароли, cookies) в БД хранить **только в зашифрованном виде** (AES-GCM с `ENCRYPTION_KEY`). В логи секреты не писать.
